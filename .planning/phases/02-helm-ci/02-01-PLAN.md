**Change 2: Init container for pg_isready (D-01, D-02, D-03, D-04)**

Add an `initContainers:` block in `spec.template.spec`, BEFORE the `containers:` block. It must be wrapped in `{{- if .Values.postgres.managed }}` so it only renders in managed mode.

Insert this block after `{{- end }}` of imagePullSecrets and before `containers:`:

```yaml
      {{- if .Values.postgres.managed }}
      initContainers:
        - name: wait-for-postgres
          image: postgres:16-alpine
          command:
            - sh
            - -c
            - |
              until pg_isready -h {{ include "clay.fullname" . }}-postgres-rw; do
                echo "Waiting for Postgres to become ready..."
                sleep 2
              done
              echo "Postgres is ready."
      {{- end }}
```

The init container host is `{{ include "clay.fullname" . }}-postgres-rw` because:
- Cluster name in cnpg-cluster.yaml is `{{ include "clay.fullname" . }}-postgres`
- CNPG creates a RW service named `{cluster-name}-rw`
- So the service is `{fullname}-postgres-rw`

**Change 3: DATABASE_URL env injection (D-09, D-10, D-11)**

Add an `env:` block to the app container, BEFORE the `envFrom:` block. This explicit `env` entry takes precedence over any same-named key in `envFrom`.

Insert this block inside the container spec, after `imagePullPolicy:` and before `ports:`:

```yaml
          {{- if .Values.postgres.managed }}
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: {{ include "clay.fullname" . }}-postgres-app
                  key: uri
          {{- else if .Values.postgres.external.dsn }}
          env:
            - name: DATABASE_URL
              value: {{ .Values.postgres.external.dsn | quote }}
          {{- end }}
```

In managed mode: secretKeyRef points to `{fullname}-postgres-app` (CNPG-generated secret for cluster `{fullname}-postgres`), key `uri`.
In external mode: plain value from `.Values.postgres.external.dsn`.

The secret name MUST be `{{ include "clay.fullname" . }}-postgres-app` (matching the Cluster name `{fullname}-postgres` + CNPG convention `-app`). If this doesn't match the Cluster name in cnpg-cluster.yaml, the pod will fail with `CreateContainerConfigError`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CNPG secret -> app container | Database credentials cross from CNPG-managed secret into app env |
| External DSN -> deployment spec | Plain-text DSN in values.yaml visible in `kubectl describe pod` |
| Init container -> CNPG RW service | Network connection from init container to Postgres service |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | deployment.yaml external DSN | accept | External DSN is a plain `env.value` visible in `kubectl describe pod`. Acceptable for hobby shop context; production users should use a Kubernetes Secret instead. Documented in RESEARCH.md. |
| T-02-02 | Spoofing | init container pg_isready | accept | Init container connects to CNPG RW service by DNS name within the cluster network. Spoofing requires cluster-level network compromise, which is out of scope for ASVS L1. |
| T-02-03 | Denial of Service | init container retry loop | mitigate | Init container loops `pg_isready` with sleep 2 indefinitely. If Postgres never starts, pod stays in Init:0/1. Kubernetes liveness/readiness probes and deployment timeout settings handle this at the orchestration layer. No additional mitigation needed — standard Kubernetes behavior. |
| T-02-04 | Information Disclosure | CNPG secret `uri` key | mitigate | Credentials stay in Kubernetes Secret, injected via `secretKeyRef` — never appear in values.yaml, ConfigMap, or Deployment spec. CNPG owns credential rotation. |
</threat_model>
