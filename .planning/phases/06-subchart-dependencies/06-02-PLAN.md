<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| values input → schema validation | User-supplied values pass through JSON Schema before helm renders templates |
| template render → validateDB guard | If user sets postgres.managed=false but omits DSN, render fails with a clear error rather than silently deploying a broken app |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Tampering | values.schema.json boolean enforcement | mitigate | `"type": "boolean"` in schema causes `helm lint` to reject strings like `"yes"`, `"1"`, `"true"` (as strings) — only actual JSON booleans pass |
| T-06-06 | Information Disclosure | clay.validateDB error message | accept | Error message reveals config structure ("postgres.managed or postgres.external.dsn required") — acceptable as this is a developer-facing Helm render error, not a runtime user-facing message |
| T-06-07 | Denial of Service | missing validateDB call site | mitigate | Plan requires adding `include "clay.validateDB"` to deployment.yaml at render time — if omitted, CI helm template tests will catch a silent misconfiguration |
</threat_model>
