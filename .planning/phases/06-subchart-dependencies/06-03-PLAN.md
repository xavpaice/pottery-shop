<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Helm registry → Chart.lock digests | Chart.lock SHA256 digests pin exact tarball content after first fetch; subsequent installs verify against lock |
| charts/ tarballs → cluster install | Tarballs are the install artifacts; they are served from the chart directory on install |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-08 | Tampering | chart/clay/charts/*.tgz | mitigate | Chart.lock SHA256 digests verify tarball integrity on `helm install`; tarballs should be committed to git or a chart repository rather than fetched at install time for reproducibility |
| T-06-09 | Denial of Service | helm dependency update network failure | accept | Helm dependency update requires network access to chart repos; if repos are unavailable, the operator install path is blocked. For air-gapped environments, this is a known post-v1.2 concern (CHART-F01) |
| T-06-10 | Repudiation | Chart.lock not committed to git | mitigate | Chart.lock must be committed to git (do not .gitignore it); this provides audit trail of which operator versions were tested and approved. The charts/ tarballs themselves may be gitignored but Chart.lock must not be. |
</threat_model>
