## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | ingress.yaml (custom mode empty secretName) | mitigate | `clay.validateIngress` fails with "ingress.tls.secretName required for custom mode" when secretName is empty -- prevents Ingress resource from referencing a nonexistent TLS Secret |
| T-03-02 | Denial of Service | ingress.yaml (letsencrypt mode empty email) | mitigate | `clay.validateIngress` fails with "ingress.tls.acme.email required for letsencrypt mode" -- prevents silent ACME registration failures |
| T-03-03 | Tampering | ingress.yaml (invalid tls.mode value) | mitigate | `clay.validateIngress` validates mode is one of letsencrypt/selfsigned/custom -- prevents typos from silently rendering a broken Ingress |
| T-03-04 | Tampering | values.yaml (nginx annotation on Traefik cluster) | mitigate | INGR-04 removes nginx annotation entirely; Traefik annotations hardcoded in template -- no operator-supplied annotations block to inject unexpected routing rules |
| T-03-05 | Information Disclosure | values.yaml (no secrets in values) | accept | This phase only references TLS secret names, never secret contents. TLS secrets are managed externally (custom mode) or by cert-manager (Phase 4). No secret data flows through values.yaml. |
</threat_model>
