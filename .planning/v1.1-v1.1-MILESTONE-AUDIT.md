## Integration Findings

### Phase 01 → Phase 02: CONNECTED ✓

`main.go` correctly consumes Phase 01 outputs: `DATABASE_URL` via `os.Getenv`, `pgxpool.New` + `stdlib.OpenDBFromPool`, `goose.SetBaseFS(migrations.FS)` + `goose.Up`. All SQL uses `$N` params and `RETURNING id`. No SQLite artifacts. CGO-free build consistent across Dockerfile and Makefile (4× `CGO_ENABLED=0`).

### Phase 02 → Phase 03: CONNECTED ✓

Ingress refactor builds cleanly on Phase 02 chart structure. Existing CI values files (`managed-values.yaml`, `external-values.yaml`) still pass `helm lint` — `ingress.enabled` defaults to `false` so `clay.validateIngress` is never called during those CI runs.

**Architecture note:** Commit `c0de261` removed the CNPG subchart dependency from `Chart.yaml`. CNPG is now a pre-installed operator (step in `integration-test.yml`), not a Helm subchart condition. Phase 02 SUMMARY.md describes the old subchart architecture but current implementation is correct.

### Phase 03 → Phase 04: UNWIRED (Phase 4 not started)

`clay.tlsSecretName` helper has a forward reference comment ("Used by both ingress.yaml and Certificate CR (Phase 4)"). For `letsencrypt` and `selfsigned` modes, `{fullname}-tls` secret name is computed but no Certificate CR exists to provision it. Ingress renders and is accepted by Kubernetes, but TLS is never provisioned.

Additionally, the `ingress.yaml` template does not render a `cert-manager.io/cluster-issuer` annotation for letsencrypt mode — cert-manager will not issue a certificate without knowing which ClusterIssuer to reference.

### CI Coverage Gap

`chart/tests/helm-template-test.sh` contains 10 behavioral tests covering INGR-01 through INGR-04 and TLS-03 (`make lint` → `make helm-test` invokes it). **Neither `test.yml` nor `integration-test.yml` call `make lint` or `make helm-test`.** Phase 03 behavioral validation runs zero times in CI.
